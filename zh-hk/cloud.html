雲服務的三种形態：SaaS、MaaS 和背后的基礎设施邏輯

雲服務的本质很简单：把計算、存儲、網絡变成像水电一样按需取用的資源。不需要买硬件、不需要等採購、不需要自己運維。

这个模式催生了三种主要的服務形態：

SaaS（Software as a Service）——软件即服務。不需要安装，打开浏览器就用。Google Docs、Notion、Figma 都是 SaaS。背后的伺服器、數據库、運維全部由提供商管理，用戶只需关注業務本身。

MaaS（Model as a Service）——模型即服務。AI 模型以 API 的形式提供。調用 OpenAI 的 API 就是在用 MaaS。不需要买 GPU、不需要训练模型、不需要部署推理服務，几行代码就拿到世界顶尖的 AI 能力。

这些服務形態有一个共同点：在本地几乎无法複現。

  • 按需取用算力：需要一台伺服器，几分钟内创建完成，用完后释放。採購周期从数月压缩到分钟级。
  • 按量付费：用多少付多少，没有闲置浪费。入門門槛从数十万降到几块钱。
  • 全球覆盖：服務可部署到全球数十个區域，用戶从最近的數據中心获取數據。
  • 托管服務：數據库、消息队列、大數據分析——开箱即用，无需自行搭建维护。

雲服務不可替代的四个维度

很多人以为雲服務的核心优势是"省钱"。这种理解过于表面。雲服務真正的不可替代性在于：有些事情本地和環境部署物理上就做不到。

  • 物理覆盖：在全球二十个區域同时部署服務，意味着在二十个地方建數據中心。这不是預算问题，是物理上不可能。雲廠商的全球基礎设施让开发者一次部署，全球可用。
  • 規模算力：AI 模型训练需要数百到数千张 GPU 并行計算。自建同等規模的集群需要数月时间、数百万资金。雲上将这一过程压缩到几分钟，用完即释放。
  • 彈性伸縮：流量在数秒内暴涨百倍并非罕见——产品上線、营销活动、病毒传播。自建伺服器无法应对这种瞬时波动。雲平台的自动扩缩可在秒级完成資源调整。
  • 合規認證：SOC 2、GDPR、HIPAA 等合規認證，企业自行获取需要数年时间和大量投入。雲廠商已具备数百项認證，用戶可直接继承。

SaaS 和 MaaS 天然是雲原生的。SaaS 需要 24/7 在线、全球可访问；MaaS 背后是大規模 GPU 推理集群。这些模式在本地環境无法落地。

从本地脚本到可調用的雲端 AI 服務

AI 自動化工具的开发是典型的从本地到雲端的遷移路径。

AutoApply 系列是基于 Playwright 和 Selenium 开发的 AI Agent，用于自動化信息筛选与投递。当前运行在本地環境，意味着需要持续开机、保持網絡穩定、无法并发运行。当一个脚本一天只能处理有限数量的任务时，它的上限就是本地硬件的上限。

AI Harness Engineering 的工作方式是多模型协作——同时調用 Claude、GPT、DeepSeek 等模型，通过提示工程编排任务邏輯。这套流程目前在本地终端中运行，本质上仍是单机工具。

遷移到雲上之后，变化是结构性的：

  • AutoApply 可打包为 Docker 镜像，部署到雲伺服器上 24/7 运行，不受本地设备状态影响
  • 可同时启动多个实例并行处理，任务量提升一个数量级
  • AI 推理网关统一调度多模型，实现自动路由、容错重试、用量统计
  • 从"自己用的脚本"变成"可調用的 API 服務"——这是向 MaaS 方向迈进的路径

SaaS 和 MaaS 的商業模式本质上是雲原生的。要做订阅制的 AI 服務，雲是唯一的选项。

去中心化的梦，中心化的伺服器

區塊鏈項目对雲基礎设施的真实依赖

區塊鏈項目在理念上是去中心化的，在基礎设施层面却高度依赖中心化的雲服務。这不是矛盾，是物理限制下的最优解。

本地运行全節點的體驗可以说明这一点：一台家用电脑同步以太坊账本需要数天，持续运行后对頻寬和存儲的占用超出多数个人環境的承受范围。行业數據显示，约三分之一的以太坊節點托管在 AWS 上——这不是項目方偷懒，是个人網絡和硬件条件根本扛不住。

Web3 項目对雲服務的依赖不止于跑節點：

  • NFT 铸造：开盘瞬间流量从零暴涨至数万并发，本地伺服器无法响应这种瞬时压力
  • DeFi 協議：需要 24/7 高可用架構和跨區域灾备
  • 链上數據分析：區塊鏈數據以 TB 级增长，分析处理必须在雲端完成
  • 交易所：全球用戶需要低延迟访问，必须依赖多區域部署

发展方向：将 Web3 基礎设施部署标准化，用 Docker 和自動化脚本实现區塊鏈節點的一键部署、自动監控与故障恢復。同时搭建从節點同步到數據清洗、入库、可视化的完整數據管道。

技術说话，账本做决定

选择雲服務背后的商業邏輯

从商業角度审视雲服務，價值比技術层面更清晰：

  • 改变成本结构：固定資產投入转为按需运营支出。资金配置更灵活，现金流压力更小。
  • 压缩交付周期：基礎设施不再成为交付瓶頸。从新服務上線到全球部署，时间单位从月缩短到天。
  • 释放团队产能：運維基礎设施的人力成本往往是隐性却最高的。将这部分工作外包给雲廠商，团队可专注于构建核心業務。
  • 打开市場范围：全球部署能力让小团队也能服務全球用戶。市場边界不再受基礎设施限制。
  • SaaS/MaaS 的商業模式：订阅制服務、按量计费的模型天然依赖雲基礎设施。如果業務方向是提供可調用的服務，雲是唯一的可行路径。

技術能力、商業判断、市場理解——三个方向相互支撑。雲服務的落地既需要技術理解，也需要商業语言将其转化为決定依据。
